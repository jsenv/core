Popover + View Transitions

No Preact/framework here — plain HTML popover API (showPopover()/hidePopover()) positioned by hand, with the open/close swap wrapped in document.startViewTransition() instead of a CSS transition/@starting-style. Click the button to open, click it again to close. Change "side" (even while open) to see it reposition, or "transition" to compare techniques — each is just a pair of `@keyframes` bound to `::view-transition-old/new(demo-popover)`.

Plain CSS transition experiments

No View Transition API here — the same aria-expanded-driven CSS transition popover.jsx itself uses (transitions suppressed, positioned, reflowed, re-enabled, then aria-expanded flips — see its own comments for why, in short: @starting-style can't see a measurement-dependent "closed" state in time). Comparing 3 recipes for a real-anchor "below" entrance, each isolating one idea. Trigger button and popover content are identical across all 3 (only the heading above each differs), so the recipe itself is the only variable when comparing them.

clip-path alone

Reads as a mask being *removed*, not an opening: the content is already at its final size underneath, just revealed.

scaleY alone

Reads as an actual opening, but content (text/icons) looks slightly squished for the first instants of the transition.

clip-path + scaleY combined

Each compensates for the other's flaw: scaleY does most of the work, a shallow clip-path (doesn't need to be aggressive — even 5-10% of the box) hides the "squished" bottom edge without clip-path's own "mask" look when used alone.
